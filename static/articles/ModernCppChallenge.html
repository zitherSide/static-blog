2.1. friend関数は演算子オーバーロードに便利

friend関数はクラス内に書くことができて、クラスのprivateメンバにアクセスできる。 でもメンバ関数ではないのでthisへの参照はない。

Instance.function() と書きたくないけど、privateメンバにアクセスしたときに使う。 OperatorのOverloadなどで利用すると良い感じ。

二項演算子のOperatorの場合は呼び出す側はあまり変わらないけど、暗黙の引数 this と仮引数 other を比べるよりも、 両方のオペランドを明示的に仮引数で受け取ったほうがわかりやすい…​と思う。

class MyClass{
private:
  int value_;
public:
  friend bool operator==(const MyClass& left, const MyClass& right){return left.value_ == right.value_;}
};

2.3. 畳み込み式

…​ で可変長テンプレート変数を畳み込み式を展開できる。 こういう記号系の文法は検索性が悪いので、畳み込み式 という名前がわかったことも良かった。

左右から畳み込み可能で …​ (op) args なら ((arg1 op arg2) op arg3) …​ と展開される

template<class ... Args>
int sum(Args... args){
  return (... + args);
}

2.4. explicitはキャストにもつけられる

explicit は引数が一つのコンストラクタにつけることで、暗黙のキャストとして扱うことを禁止できるが、 キャストにもつけられる。

コンストラクタにしか使ったことないけど、キャストを自分で定義してそこそこ計算量がある場合は暗黙に実行されないように explicit をつけられるみたい。

using namespace std;

class MyClass{
public:
    int value_ = 10;
    explicit operator int() const{
        return value_;
    }
};

int main(){
    cout << (int)MyClass() << endl; //OK
    cout << MyClass() << endl; //NG
    return 0;
}

2.5. リテラル演算子もオーバーロードできる

1u とか 0.1f みたいにリテラルの後ろに書く u とか f はリテラル演算子と呼ばれる。 これもオーバーロードできる。

浮動少数リテラルにつけて使うリテラル演算子は、引数を long double の型として受け取る

  return quantity<scale::kelvin>(static_cast<double>(amount));

constexpr quantity<scale::fahrenheit> operator ""_k(long double const amount) {

2.6. io_base::setf で書式設定できる

iostream で表示するときに数値の0詰めなどの書式を設定できる。 これは io_base にあるけど setw は iominip にある。

boolを文字列表示できる boolalpha は ios にある。

…​なぜこんなにばらけているのだろう…​？

バイト配列を16進表記文字列にするサンプル。setfで大文字化、setwとsetfillで2文字の0詰めにしている。

#include <iostream>
#include <iomanip>
#include <vector>

using namespace std;

template<typename Iter>
string bytes_to_hexstr(Iter begin, Iter end, bool const uppercase = false) {
	ostringstream oss;
	if (uppercase) oss.setf(ios_base::uppercase);
	for (; begin != end; ++begin)
		oss << hex << setw(2) << setfill('0') << static_cast<int>(*begin);
	return oss.str();
}

template<typename C>
string bytes_to_hexstr(C const& c, bool const uppercase = false) {
	return bytes_to_hexstr(cbegin(c), cend(c), uppercase);
}

int main(){
    cout << bytes_to_hexstr(vector{0, 1, 14, 5}, true) << endl;
    return 0;
}

2.7. char_traitsとallocator を使うことで独自のstring型を作れる

char_traits は basic_string で利用する型情報や関数を備えたトレイト。 char , char8_t , wchar_t など基本的なchar型に対して特殊化されたものが定義済み。

allocator はメモリ確保のトレイト。要するに普通にnewしてくれる。 自分でメモリプールなど用意して、そこから確保してほしいときなどに、こんな感じで自分で allocator トレイトを実装して使う。

vector<int, PoolAllocator<int>> pooledVector;

本書では独自のstring型を作るのではなく、任意の base_string から派生した型に対する split 関数を定義するために利用していた。

template<class Elem>
using tstring = basic_string<Elem, char_traits<Elem>, allocator<Elem>>;

template<class Elem>
using tstringstream = basic_stringstream<Elem, char_traits<Elem>, allocator<Elem>>;

template<class Elem>
inline vector<tstring<Elem>> split(tstring<Elem> text, Elem const delimiter) {
	auto sstr = tstringstream<Elem>{ text };
	auto tokens = vector<tstring<Elem>>{};
	auto token = tstring<Elem>{};
	while (getline(sstr, token, delimiter)) {
		if (!token.empty()) tokens.push_back(token);
	}
	return tokens;
}

template<class Elem>
inline vector<tstring<Elem>> split(tstring<Elem> text, tstring<Elem> const& delimiters) {
	auto tokens = vector<tstring<Elem>>{};
	size_t pos, prev_pos = 0;
	while ((pos = text.find_first_of(delimiters, prev_pos)) != tstring<Elem>::npos) {
		if (pos > prev_pos)
			tokens.push_back(text.substr(prev_pos, pos - prev_pos));
		prev_pos = pos + 1;
	}
	if (prev_pos < text.length())
		tokens.push_back(text.substr(prev_pos, tstring<Elem>::npos));
	return tokens;
}

テンプレートパラメータ Elem をもとに char_traits を作ってbase_stringにわたす型をわざわざ定義している。 base_string がスーパークラスなのだから、そのまま下記のようにbase_stringを引数や戻り値としてIFとして書いてしまえば良いと思うが、 そうすると、delimiterとtextの実体の型があっていなくてもコンパイルできてしまったり、戻り値の実体の型を決めるために関数内でtextの型を動的に調べなければいけなくなってしまう。

2.9. 正規表現のマッチを表すイテレータ *regex_iterator

正規表現検索は複数の結果がマッチする。その結果を示すイテレータ。 match と同じようにそれぞれの型に対応するエイリアスが定義済み。

イテレータの終端は普通のコンテナだと vec.cend() みたいに取得するが、正規表現マッチのイテレータは 引数なしで作成したイテレータ sregex_iterator() が終端イテレータになる。

regex rx(R"(([A-Z]{3}-[A-Z]{2} \d{3,4})*)");
smatch match;
vector<string> results;

for (auto i = sregex_iterator(cbegin(str), cend(str), rx); i != sregex_iterator(); ++i) {
	if ((*i)[1].matched)
		results.push_back(i->str());
}
return results;

2.13. テンプレート引数で受け取った呼び出し可能なオブジェクトは実際の型によらず invoke で実行できる

関数、メンバ関数、function型の変数、関数オブジェクト(Functor)、などC++では Foo() の形で呼び出せるオブジェクトが何種類も存在する。 これらを一様に同じ操作で呼び出すのが invoke 。本書では処理時間を計測するためのヘルパー関数で、任意の呼び出し可能なオブジェクトを設定できるようにするために使っていた。

template<typename Time = microseconds, typename Clock = high_resolution_clock>
struct perf_timer {
	template<typename F, typename... Args>
	static Time duration(F&& f, Args... args) {
		auto start = Clock::now();
		invoke(forward<F>(f), forward<Args>(args)...);
		auto end = Clock::now();

		return duration_cast<Time>(end - start);
	}
};

auto t1 = perf_timer<microseconds>::duration(f);
cout << "t1: " << t1.count() << endl;

2.15. スタンダードレイアウトな型、標準レイアウトな型

ざっくりとメンバの宣言順にメモリ上に配置される型のこと。以下のことなどをするとスタンダードレイアウトでなくなる

  • 仮想関数をもつ

    • v_tableが挿入されるのでメモリ配置がかずれる

  • 継承する

    • スーパークラスのメンバが挿入されるのでメモリ配置がずれる

    • EBO (Empty Base Optimization)が効くので空のクラスの継承ならコンパイラはメモリ上に何も挿入しないのでスタンダードレイアウトのままになる

      • C++のオブジェクトは絶対にアドレスが取得できなければならないので、普通は殻のオブジェクトでも何らかのメモリ上のサイズを専有する。EBOとは継承すると継承先のデータがメモリ上に確保されるようになるのでアドレスを割り当てるために空のクラスのダミーの領域を専有する必要がなくなり、スーパークラスのメモリ上のサイズが０になること

標準レイアウトだとかならず連続領域に確保されたり、オブジェクトのアドレスと先頭メンバのアドレスが一致したりするが、 それによって何が嬉しいのかはよくわかっていない…​

多分メモリ領域を直接操作するようなアルゴリズムに関数を特殊化するときや、他の言語とのABIを守らなければ行けない関数の concept とかで利用するのかな？

2.18. make_heap push_heap pop_heap でヒープ木構造(優先度付きキュー)が作れる

ここでいうヒープ木はデータ構造の話で、 new したりするときに確保されるヒープ領域の話ではない。 (単純に英単語として heap: 積み重ねたもの という意味で「積み重ねた木」と「動的確保のために積み重ねたメモリ領域」と名付けたら同じ名前になっただけ)

ヒープ木作成の優先度を決定する Compare 関数を渡すと、任意のコンテナの中身をヒープ木として構築してくれる。s

template<class T, class Compare = std::less<typename std::vector<T>::value_type>>
class priority_queue
{
	using value_type = typename std::vector<T>::value_type;
	using size_type = typename std::vector<T>::size_type;
	using reference = typename std::vector<T>::reference;
	using const_reference = typename std::vector<T>::const_reference;

private:
	std::vector<T> data;
	Compare comparer;

public:
	bool empty() const noexcept { return data.empty(); }
	size_type size() const noexcept { return data.size(); }
	void push(value_type const & value) {
		data.push_back(value);
		std::push_heap(std::begin(data), std::end(data), comparer);
	}
	void pop() {
		std::pop_heap(std::begin(data), std::end(data), comparer);
		data.pop_back();
	}
	const_reference top() const { return data.front(); }
	void swap(priority_queue& other) noexcept {
		swap(data, other.data);
		swap(comparer, other.comparer);
	}
};

2.19. 自作のコンテナのiteratorの種類をiterator_tagとして定義することで、既存のSTLの実装で適切に特殊家してくれる

空のクラス iterator_tag を継承した、これまた空のクラスとして

  • input_iterator_tag

  • output_iterator_tag

  • forward_iterator_tag

  • backward_iterator_tag

  • random_access_iterator_tag

などがある。空のクラスなので、これ自体が何かを実装する訳では無いが、distance() などイテレータのUtil実装の際に、 iterator_tag をテンプレートパラメータとして受け取り、特殊化の際にSwitch文的にこれらの型を使ってイテレータアクセスの実装を切り替えている。

自作のコンテナ用に自作のイテレータを作成する場合、この型を定義しておけばSTLの既存の関数からも利用できる。

template<class T>
class circular_buffer_iterator {
	using self_type = circular_buffer_iterator;
	using value_type = T;
	using reference = T & ;
	using const_reference = T const&;
	using pointer = T * ;
	using const_pointer = T const*;
	using iterator_category = std::random_access_iterator_tag;
	using difference_type = ptrdiff_t;
};

2.26. イテレータ破壊を防ぐために挿入イテレータを利用する

イテレータ破壊とは、イテレータ取得後にコンテナに要素追加や要素削除を行った結果、取得していたイテレータが無効な場所を示している状態になること。 よくやるのがfor文中で条件に一致したものを削除しようとしてイテレータ破壊を起こして、要素の探索に漏れが出てしまう場合。

下記の vector から 0 の要素を取り除くfor文は正しく動かない。 {1, 0, 2, 0, 0, 4} から０を消して {1, 2, 4} となることを期待するが、 erase でイテレータ破壊が起こるので　{1, 2, 0, 4} となってｓまう。

vector<int> vec{1, 0, 2, 0, 0, 4};
for(auto ite = vec.begin(); ite !=vec.end(); ++ite){
	if(*ite == 0)
		vec.erase(ite); //vectorの要素が消えてイテレータが一つ進んでしまうので、for文の ++ite と合わせて２つ進んでしまう
}

削除するときは erase_if や remove_if → erase を呼ぶことでイテレータ破壊を回避するが、挿入する場合は挿入イテレータを利用する。 挿入イテレータは、イテレーターに代入すると、イテレータの示す先のコンテナに挿入する操作に変換してくれるものだ。 例えば vector の末尾挿入イテレータ( back_inserter_iterator )で代入を繰り返すと、vector の末尾に要素がどんどん追加される。 stream みたいな使用感。

int main(){
    vector<int> vec{};
    auto ite = back_inserter(vec);
    *ite = 5;
    *ite = 6;

	//vec = [5, 6]
    copy(vec.cbegin(), vec.cend(), ostream_iterator<int>(cout, ", "));
    return 0;
}

ちなみに ++ しても意味はない。実際に代入しないことにはイテレータが進むことはない。 copy などのコンテナを受け取る関数に渡して利用することが多い。

文字列を一文字づつ unsinged char のvectorとして保持するようにコピーする例。

std::vector<unsigned char> from_string(std::string_view data) {
	std::vector<unsigned char> result;
	std::copy(std::begin(data), std::end(data), std::back_inserter(result));
	return result;
}

2.27. std::decayによってテンプレート引数として渡された型を取り扱いやすい型に変換できる

難しくて分かりづらいので、まずはサンプルコードを見る。

template<typename T, typename A, typename F,
	typename R = typename decay_t<
		typename result_of_t<
			typename decay_t<F>&(typename vector<T, A>::const_reference)
		>
	>
>
vector<R> select(vector<T, A> const& c, F&& f){
	vector<R> v;
	transform(cbegin(c), cend(c), back_inserter(v), forward<F>(f));
	return v;
}

この関数は「もとのコンテナ c から任意の値をうみだす関数 F を通して得られる結果を保持するコンテナを得る」関数だ。 機能としてはtransformそのもので、 begin , end のイテレータを渡す代わりにコンテナを渡せばOK、というtransformのラッパーだ。

transformをラップするときに decay が登場している。戻り値の型 vector<R> が F の戻り値を decay した型だ。 なぜわざわざ decay しなければいけないのか、を理解することが decay の役割を理解することになる。

まずは decay の動きを確認する。 decay は remove_cv + remove_reference + 配列のポインタ化 + 関数のポインタ化 or function型化 だ。

#include <type_traits>

using namespace std;

int func(int) { return 0;}

int main(){
    int integer;
    int& ref_int = integer;
    int arr[3];

    cout << typeid(decay_t< decltype(integer) >).name() << endl; //int （そのまま)
    cout << typeid(decltype(integer)).name() << endl;			 //int
    cout << "----" << endl;

    cout << typeid(decay_t< decltype(ref_int) >).name() << endl; //int (remove_referenceされる)
    cout << typeid(decltype(ref_int)).name() << endl;			 //int (int&かと思ったら普通にintだった...なぜ？)
    cout << "----" << endl;

    cout << typeid(decay_t< decltype(arr) >).name() << endl;     //int* (array -> pointer化)
    cout << typeid(decltype(arr)).name() << endl;				 //int[3]
    cout << "----" << endl;

    cout << typeid(decay_t< decltype(func) >).name() << endl;    //int(*)(int) (function -> function pointer化)
    cout << typeid( decltype(func) ).name() << endl;			 //int()int

    return 0;
}

まぁ要するに「テンプレート引数でもらった型が const だったり参照だったり、関数そのものだったり配列だったりで、そのままでは代入できなかったり 変更できなかったりする型を全部もとの型に戻してあげましょう」ということ。

最初のコードでは戻り値の型 vector<R> のために typename R = typename decay_t<typename result_of_t<typename decay_t<F>&(typename vector<T, A>::const_reference)>> で型修飾を外している。読みづらいので typename と const_reference を省けば decay_t<result_of<decay_t<F>&(vector<T, A>)>> つまり「関数 F の戻り値の型」だが、Fが関数そのものだったり関数ポインタだったりするのを扱いやすい関数ポインタに統一するために decay して、その戻り値の型が参照だったり配列だったりして vector の要素になれないかもしれないから decay する、ということ。

ちなみに decay_t は他のメタプログラミング用関数と同じく decay<>::type のエイリアス。 そして decay は崩壊する、とか腐るといった意味。「何かがボロボロと剥がれ落ちて中身が出てくる」みたいなニュアンスなので、 もとの複雑に修飾された型を単純な型に戻す関数の名前に採用したんだと想像している。

Note
 サンプルコードの result_of メタ関数は invoke の導入に合わせて命名規則を統一するために invoke_result に置き換えられた

2.32. 並列処理のスレッド数の目安として thread::hardware_concurrency が利用できる

hardware_concurrencyで取得できたスレッド数でコンテナに対する任意の繰り返し処理 F を並列化する例。 MapReduce処理をしており、Mapした処理結果をFutureとして保持して、最後にReduceする。 std::async で呼び出すことで結果は future となって非同期で実行できる。

また、iteratorからもとのコンテナに関する型情報を取得するのに iterator_traits を利用している。

template<typename Iterator, typename F>
auto aprocess(Iterator begin, Iterator end, F&& f) {
	auto size = distance(begin, end);
	if (size <= 10000)
		return forward<F>(f)(begin, end);
	else {
		int const task_count = thread::hardware_concurrency();
		vector<future<typename iterator_traits<Iterator>::value_type>> tasks;

		auto first = begin;
		auto last = first;
		size /= task_count;
		for (int i = 0; i < task_count; ++i) {
			first = last;
			if (i == task_count - 1)
				last = end;
			else
				advance(last, size);

			tasks.emplace_back(
				std::async(
					launch::async, [first, last, &f]() {
						return forward<F>(f)(first, last);
					}
				)
			);
		}
		vector<typename iterator_traits<Iterator>::value_type> mins;
		for (auto & t : tasks)
			mins.push_back(t.get());

		return forward<F>(f)(std::begin(mins), std::end(mins));
	}
}

2.35. 順番のある処理を準備側と処理側で非同期に実行するときに condition_variable が利用できる

非同期にタスクが同じ共有資源をさわるとき、それぞれのタスクに順番がある場合がある。 例えば片方のグループはキューにジョブを詰め込んで、もう片方のグループはキューに入ったジョブを片付ける、といった場合など。

後段にあるタスクが、共有資源であるキューのmutexを確保して、キューにジョブがあるかどうかポーリングしても実現できるが、 明らかに無駄だ。

キューにジョブを詰め込む側のタスクが、詰め込んだことを通知して、それを起点に後段のタスクがキューをチェックしに行けば良い。 これを実現するのが condition_variable だ。

condition_variable は共有資源のmutexと一緒に利用する。通知をとばす側は単純に通知をとばすだけだ。 受け取る側はmutexをロックした状態で condition_variable::wait(lock, conditionFunc) をする。

するとwait関数は通知が来て、conditionFunc の条件が整うまで lock として渡したmutexのロックを外して、 通知側が処理を続行できるようにした上で待ち構える状態に入る。

通知がきて conditionFunc の状態を満たせば、再びロックを掛けてそれ以降の処理を続行する。 (本で紹介されていたコードでは共有資源であるcustomersへpushするときにロックを取っていなかった。ロックが必要だと思うので追加した。)

priority_queue<customer> customers;
bool office_open = true;
mutex mt;
condition_variable cv;

//前段タスク。 customerキューにジョブを投げる
thread store([&office_open, &customers, &cv]() {
	ticketing_machine tm(100);
	random_device rd;
	array<int, mt19937::state_size> seed;
	generate(begin(seed), end(seed), ref(rd));
	seed_seq seq(cbegin(seed), cend(seed));
	mt19937 eng(seq);
	uniform_int_distribution ud(200, 500);

	for (int i = 1; i < 25; ++i) {
		customer c(tm.next());
		{
			unique_lock<mutex> lock(mt);
			customers.push(c);
		}
		cv.notify_one();
		this_thread::sleep_for(chrono::milliseconds(ud(eng)));
	}
	office_open = false;
});

//後段タスク customerが追加された通知を受けて、処理を始める
vector<thread> desks;
for (int i = 0; i <= 3; ++i) {
	desks.emplace_back([i, &office_open, &mt, &cv, &customers]() {
		random_device rd;
		array<int, mt19937::state_size> seed_data;
		generate(begin(seed_data), end(seed_data), ref(rd));
		seed_seq seq(cbegin(seed_data), cend(seed_data));
		mt19937 eng(seq);
		uniform_int_distribution ud(2000, 3000);


		while (office_open || !customers.empty()) {
			unique_lock<mutex> lock(mt);

			cv.wait_for(lock, chrono::seconds(1), [&customers]() { return !customers.empty(); });

			if (!customers.empty()) {
				auto const c = customers.top();
				customers.pop();

				lock.unlock();
				cv.notify_one();

				this_thread::sleep_for(chrono::milliseconds(ud(eng)));
			}
		}
	});
}



store.join();
for (auto & desk : desks)
	desk.join();

	m.year = element.at("year").get<unsigned int>();
	m.length = element.at("length").get<unsigned int>();
	for (auto const & role : element.at("cast")) {
		m.cast.push_back(casting_role{
			role.at("start").get<string>(),
			role.at("name").get<string>()
			});
	}
	for (auto const& writer : element.at("writers")) {
		m.writers.push_back(writer);
	}
	movies.push_back(move(m));
}

nlohmanjson::json 型は stream から >>, << できるので、ファイル読み書きなら fstream ←→ json ←→ T という感じでやり取りする。

2.49. tellg, tellp, seekg, seekp

tellg, seekg は tell get seek get であり読み取り位置の取得と移動、 tellp, seekpは tell put, seek put で書き込み位置の取得と移動。

ファイルストリームを扱っている場合は、読み込み位置と書き込み位置は同じなのでどちらを使っても同じ動作になる。

2.51. ハッシュ値からもとのパスワードを類推することを難しくするためにパスワードにソルトを付与する

パスワードを平文で保存すると漏洩したときにまずい。この問題を回避するために、パスワード自体は保存せず、 パスワードのハッシュ値を保尊する。

パスワードが入力されるとハッシュを計算し、それが登録されたハッシュと一致するかどうかで認証を行う。 ハッシュ値であれば漏洩したとしてももとのパスワードを復元することはできない。

確かにできないけど、調べることはできてしまう。よくあるパスワードを入力してハッシュを計算させれば、 そのハッシュを検索して、もとパスワードを使っている人を特定できる。

この問題に対処するのがソルト。パスワードごとにランダムな文字列を付与してからハッシュ計算をさせることで、 同じパスワードでも違うハッシュが出てくるようになって、もとのパスワードはわからない。

ただ、この方法では認証時にパスワードに同じソルトを与えてハッシュを計算する必要があるので、 どのパスワードがどのソルトを使ってのかも覚えておく必要がある。 そっちも一緒にバレてしまうとよくあるパスワードとソルトを総当りしてみる、ということができなくもない。 (ソルトがない場合よりずっと大変だけど)

2.52. 署名検証と暗号復号

どちらも公開鍵と秘密鍵で暗号化と復号を行う。 公開鍵で暗号化するのが普通の暗号で、秘密鍵で暗号化するのが署名。

署名の場合は元ファイルから生成したハッシュを秘密鍵で暗号化する。 検証者は公開鍵で復号したハッシュと、自分で元ファイルから生成したハッシュを比べることで、 ファイルが改ざんされておらず、作成者（の秘密鍵)が同一であることを確認できる。

2.53. thisポインターを shared_ptr にできる enable_shared_from_this

this ポインターから shared_ptr を作ってはいけない。 shared_ptr を this から作ると、その時点から this への参照カウントが増える。 しかし this はもともと自分のインスタンスが参照している「生ポインター」なので自分の分をカウントしていない。

正しくは shared_ptr を this から作ったときは、 「thisを持つインスタンス」と「今作ったshared_ptr」で参照カウントは 2 から始まらなければならないが、「thisをもインスタンス」が勘定されず、1 少ない状態で始まってしまう。

このせいで shared_ptr 側をリリースしたときに、まだもとのインスタンスが this を握っているのに、参照カウントが 0 になってしまって、デストラクタが実行されてしまう。

enable_shared_from_this を継承したクラスで shared_from_this() 関数を呼び出せば、このあたりの問題をクリアした状態で this をさす shrared_ptr を作れる。

ちゃんと動くためには、 this をもつインスタンス自身が shared_ptr になっている必要がある。 (thisからshared_ptrを増やすということは、もとのインスタンスもshrared_ptrになっていないと、そもそもあるポインタは生ポインタで あるポインタはshared_ptrで、という管理になってしまうので当然だが…​)
